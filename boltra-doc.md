# Boltra: FastAPI Development Kit with Rust-Powered Performance Modules

## 1. Project Vision

Boltra is a FastAPI-based development kit for building modern backend systems faster.

In the early versions, Boltra will not replace FastAPI. Instead, it will generate direct FastAPI code and provide a Django-like development experience around FastAPI.

The goal is simple:

Use FastAPI for API development now.
Use Boltra to add structure, CLI automation, apps, ORM, admin, auth, workers, AI tools, Docker, tests, and future Rust-powered performance modules.

Boltra should help developers build:

* APIs
* SaaS products
* AI applications
* admin dashboards
* CRUD systems
* backend services
* startup MVPs
* enterprise-ready projects

The early identity of Boltra is:

Boltra is a FastAPI development kit that gives FastAPI projects Django-like structure and batteries-included productivity.

## 2. Project Name

Recommended name:

Boltra

Why this name is good:

* short
* easy to remember
* unique and brandable
* related to speed and power
* suitable for a backend framework/toolkit
* can later represent Rust-powered performance
* simple for CLI usage

Example CLI:

```bash
boltra new hello
boltra dev
boltra add app school
boltra add orm
boltra add admin
```

In early versions, generated projects should use direct FastAPI code.

Generated `main.py` should use:

```python
from fastapi import FastAPI
```

not a fake upper wrapper.

## 3. Main Tagline

Boltra: Django-like productivity for FastAPI projects.

Alternative taglines:

Boltra: Build FastAPI apps faster.

Boltra: FastAPI structure, automation, and batteries included.

Boltra: Start with FastAPI, grow like Django.

Boltra: A FastAPI kit for SaaS, AI, admin, auth, and scalable backend development.

Boltra: FastAPI today, Rust-powered performance tomorrow.

## 4. Core Philosophy

Boltra should follow this philosophy:

Use direct FastAPI code.
Do not hide FastAPI.
Add missing batteries around FastAPI.
Keep the starting project small.
Let the project grow step by step.
Make every feature removable.
Add Rust performance only where it is useful later.

Boltra should not pretend to be a full independent framework in the beginning.

In early versions, Boltra should be honest:

It is a FastAPI development kit.

It helps developers build organized, scalable, production-ready FastAPI projects.

## 5. Why This Direction Is Practical

Building a full web framework from zero is very hard.

If Boltra tries to build its own API system from day one, it must build:

* routing
* request parsing
* path parameters
* query parameters
* request body parsing
* validation
* OpenAPI docs
* Swagger UI
* middleware system
* error handling
* async server integration

This will take a lot of time.

Instead, Boltra should use FastAPI for the API layer and focus on the missing parts that FastAPI developers usually build manually.

These missing parts are:

* project generator
* app/module system
* settings management
* add/remove CLI
* ORM layer
* admin panel
* auth system
* permission system
* worker system
* AI app structure
* payment structure
* Docker setup
* tests setup
* benchmark tools
* future Rust-powered performance modules

This makes Boltra useful much earlier.

## 6. Main Identity

Boltra is not a FastAPI replacement in early versions.

Boltra is a FastAPI development kit.

The generated project should use normal FastAPI code.

Example:

```python
from fastapi import FastAPI

app = FastAPI(
    title="Hello API",
    version="0.1.0",
)

@app.get("/")
async def home():
    return {"message": "Hello from FastAPI + Boltra Kit"}
```

This is important because developers can understand the project immediately.

They are not locked into a hidden wrapper.

They can use FastAPI normally.

Boltra simply gives them better structure and batteries.

```python

    "apps.school",

```

Generated `apps/school/api.py`:

```python
from fastapi import APIRouter

router = APIRouter(
    prefix="/school",
    tags=["school"],
)

@router.get("/students")
async def list_students():
    return [
        {"id": 1, "name": "Ali"},
        {"id": 2, "name": "Sara"},
    ]
```

Then `main.py` should be updated:

```python
from fastapi import FastAPI
from apps.school.api import router as school_router

app = FastAPI(
    title="Hello API",
    version="0.1.0",
)

app.include_router(school_router)

@app.get("/")
async def home():
    return {"message": "Hello from FastAPI + Boltra Kit"}
```

This is direct FastAPI.

Boltra only automates the structure.

## 14. Remove App System

Command:

```bash
boltra remove app school
```

Boltra should ask:

```text
This will remove the school app and unregister it from settings.py.
Do you want to continue? [y/N]
```

Actions:

* remove `apps/school/`
* remove `"apps.school"` from `INSTALLED_APPS`
* remove router import from `main.py`
* remove `app.include_router(school_router)`
* show summary of changes

Remove should be safe by default.

## 15. Router Auto-Discovery

Manual router imports can become messy in large projects.

Boltra should later support auto-discovery.

Example:

```python
from fastapi import FastAPI
from boltra.apps import load_apps

app = FastAPI(
    title="Hello API",
    version="0.1.0",
)

load_apps(app)

@app.get("/")
async def home():
    return {"message": "Hello from FastAPI + Boltra Kit"}
```

`load_apps(app)` should:

* read `INSTALLED_APPS`
* import each appтАЩs `api.py`
* find `router`
* include the router in FastAPI
* show clear errors if router is missing

This keeps direct FastAPI code but reduces manual work.

## 17. ORM Strategy

Boltra ORM should be built slowly.

Do not try to build a full advanced ORM first.

Start with:

* Model base class
* Integer field
* String field
* Boolean field
* DateTime field
* create
* get
* filter
* update
* delete
* count
* exists
* all
* first
* order_by
* limit
* offset

Later add (each as its own gated phase in `doc/plan/phase.md`):

* Rust SQL query builder (replace Python compiler hot path)
* Rust row decoder (replace Python deserialization hot path)
* Boltra-native migrations
* relationships (FK, select-related)
* transactions and savepoints
* joins and aggregates
* indexes and constraints
* raw SQL escape hatch

The ORM is Boltra's core differentiator тАФ async-first Python API with Rust where it measurably wins. Build slowly; never ship a layer without tests and benchmarks.

## 28. Future Rust Role

In early versions, Boltra should not depend heavily on Rust.

First, build a useful FastAPI development kit.

Later, Rust can be added for performance-critical modules.

Possible Rust modules:

* fast JSON serializer
* SQL query builder
* database row decoder
* validation engine
* password/security helpers
* compression helpers
* static file optimizer
* benchmark engine
* native API engine later

The rule:

Do not add Rust just for marketing.

Add Rust only where benchmarks prove it helps.

## 29. Future Native API Engine

Boltra may build its own native API engine later.

But this should not happen early.

Early versions should use direct FastAPI.

Future native engine goals:

* custom router
* request parser
* response serializer
* validation engine
* middleware pipeline
* OpenAPI generator
* optional FastAPI compatibility layer

Important rule:

User projects should not break when Boltra introduces a native engine.

So early Boltra should keep project structure clean and avoid unnecessary lock-in.

## 38. Critical Architecture Decisions

These decisions should be made **before Phase 10 (ORM Design)**. Document the choice in ADRs (Architecture Decision Records).

### 38.1 ORM Strategy (Decided: From-Scratch, Async-First, Rust-Powered)

Boltra ORM is a **first-class product** тАФ built from scratch, not wrapped around SQLAlchemy or SQLModel.

**Core principles:**

* **Async-first** тАФ all public APIs are `async`; sync wrappers are never the default
* **Python orchestration, Rust acceleration** тАФ connection lifecycle, model registry, and developer-facing API in Python; SQL building, parameter binding, and row decoding in Rust where benchmarks justify it
* **Slow layers, high quality** тАФ each ORM layer ships with tests, benchmarks, and docs before the next layer starts
* **Python fallback always** тАФ every Rust module must degrade gracefully if the native extension is unavailable

**Layer architecture:**

```text
Developer API (Python)     Model, Field, QuerySet, Manager
        тЖУ
Query Compiler (Python)    Validates filters, ordering, limits
        тЖУ
SQL Engine (Rust)            Query builder, identifier quoting, param binding
        тЖУ
Driver Bridge (Python)       asyncpg (PostgreSQL first)
        тЖУ
Row Decoder (Rust)           Fast tuple/dict тЖТ model instance mapping
```

**Database support order:** PostgreSQL (asyncpg) тЖТ SQLite (aiosqlite) for local dev only.

**No third-party ORM dependency** in the runtime path. Migrations are Boltra-native (not Alembic).

**Detailed execution plan:** see [`doc/plan/phase.md`](doc/plan/phase.md).

### 38.2 Settings: Pydantic Settings Early

Move `.env` support from тАЬlaterтАЭ to **Phase 7**:

```python
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    app_name: str = "hello"
    debug: bool = True
    secret_key: str
    database_url: str | None = None

    model_config = {"env_file": ".env"}
```

Keep `settings.py` as the user-facing file; CLI generates Pydantic-based settings from day one.

### 38.3 Code Generation vs AST Editing

Define rules for CLI mutations:

| Operation | Method |
|-----------|--------|
| New files / folders | Jinja2 templates |
| `INSTALLED_APPS` updates | AST parse + safe insert (libcst or ast) |
| `main.py` router imports | Prefer `load_apps(app)` over repeated AST edits |
| Idempotency | Running `boltra add orm` twice must not duplicate code |

**Rule:** After Phase 8 (auto-discovery), minimize direct `main.py` edits.

### 38.4 Single Package First, Split Later

Section 27 lists many packages (`boltra-cli`, `boltra-orm`, тАж). For v0.x:

* Ship one **`boltra`** PyPI package with internal submodules
* Split into optional extras: `boltra[orm]`, `boltra[admin]`, `boltra[ai]`
* Only extract separate repos when API boundaries are stable

### 38.5 Compatibility Matrix

Document and test against:

* Python **3.11+** (3.12 recommended)
* FastAPI **0.110+**
* Uvicorn **0.27+**
* PostgreSQL 15+ (SQLite for local dev only)

## 48. Final Summary (One Paragraph)

Boltra starts as an honest FastAPI development kit: direct `FastAPI()` code, Django-like apps and settings, and a CLI that adds/removes batteries over time. Its differentiator is a **from-scratch, async-first ORM** with **Rust-powered** SQL building and row decoding тАФ developed slowly in layered phases (see `doc/plan/phase.md`). v1 success depends on **project generation, app modularity, ORM quality, admin, and auth**. The native API engine, full enterprise suite, and payments stay deferred until the kit and ORM are stable and benchmarked.
